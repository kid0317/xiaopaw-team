# XiaoPaw-Team 详细流程设计（v0.3）

> **本文档**：在 v0.2 架构基础上，以**超细粒度**把一次完整项目从"共创 SOP"到"进化交付"的全流程讲清楚。
> **前置阅读**：v0.2《XiaoPaw-Team 详细设计文档》（架构/Tools/Skills 分层）
> **版本**：v0.3（详细流程增补版）
> **最后更新**：2026-04-20

---

## 再次强调：飞书是 Manager 的专属通道

> **架构硬约束**（Python 代码层面保证，不是 LLM 自律）：
> - 飞书的所有入站消息 → 只进 Manager Agent（Runner 路由 `p2p:ou_xxx` → `team:manager`）
> - 飞书的所有出站消息 → 只由 Manager 发出（`SendToHumanTool` 只注入 Manager）
> - PM / RD / QA 的 Crew 构造器不持有 FeishuSender 对象、不接受任何飞书 routing_key
> - 任何"团队内角色想跟用户说句话"的场景 → 只能发邮件给 Manager，由 Manager 翻译成飞书卡片转达
>
> 本文档后面所有提到"用户收到通知"、"用户回复"的段落，都是**飞书 ↔ Manager**，绝对没有飞书 ↔ PM/RD/QA 的路径。

## 阶段 0 · SOP 共创（一次性）

**发生时机**：第一次使用 xiaopaw-team 时，或当用户想为新类型项目建立新 SOP 时。一旦共创出来，后续同类项目复用，**不必每次重做**。

### 0.1 用户发起

用户在飞书发："**我们来建一个开发的 SOP 吧，以后做小功能都按这个走。**"

### 0.2 消息路径

- FeishuListener 收到消息 → `feishu_bridge.recv_human_response` 先过一遍
- 读 `checkpoints.jsonl`，发现**没有 pending checkpoint**（这是新对话）
- 构造 `InboundMessage(routing_key="team:manager", text=..., meta={"source":"feishu", "user_open_id":"ou_xxx"})`
- 进 Runner 的 `team:manager` 串行队列 → Manager Agent kickoff

### 0.3 Manager 的第一反应：意图识别

Manager Agent 的 `backstory` 里（来自 `workspace/manager/agent.md`）有**入站消息分流规则**：

```
收到用户消息后，先判断意图：
- 包含"SOP"/"流程"/"标准"等关键词 → load_skill("sop_cocreate")
- 是新开发需求（描述要做的产品/功能）→ load_skill("sop_selector")
- 是 checkpoint 回复（meta.pending_checkpoint 有值）→ load_skill("handle_checkpoint_reply")
- 是对已有 SOP 的修改意见 → load_skill("sop_cocreate") 并说明是修改
- 其他（自由提问、闲聊）→ 直接回答，不触发任何 SOP
```

这段分流规则是**用自然语言写在 agent.md 里**的，LLM 读完知道该做什么。Python 代码里没有对应的 if/else。

### 0.4 Manager 加载 sop_cocreate Skill（参考型）

Manager LLM 判断意图 = SOP 共创，调用 `SkillLoaderTool` 加载 `sop_cocreate`：

```
load_skill 调用 → SkillLoader 读 manager/skills/sop_cocreate/SKILL.md
→ 返回 SKILL 内容作为 context 给 Manager LLM（reference 类型不启动 Sub-Crew）
```

**sop_cocreate SKILL.md 的核心内容**（参考型，描述共创对话框架）：

```markdown
---
name: sop_cocreate
type: reference
---

# SOP 共创对话框架

目标：和用户一起把"某类项目的标准流程"固化为一份 SOP。
产出：一份 SKILL.md 放在 workspace/manager/skills/sop_{name}/SKILL.md

## 追问顺序（每次对话覆盖一个维度）

### 维度 1：项目类型识别
- 这类项目的典型需求是什么？（举 1-2 个例子）
- 它和现有 SOP 的区别？

### 维度 2:团队构成
- 哪些角色要介入？（Manager 必在，PM/RD/QA 可选）
- 每个角色负责什么阶段？

### 维度 3:阶段清单
- 需要哪些阶段？（典型：澄清/设计/评审/实现/测试/交付）
- 每个阶段谁是 Owner？产物是什么？

### 维度 4:checkpoint 位置
- 哪些节点必须用户拍板？
- 拍板的依据是什么？（交付物摘要）

### 维度 5:团队内评审
- 哪些阶段产物要其他角色评审？
- 评审失败的循环上限？

## 追问策略
- 每次只问 1-2 个问题，不要一次抛出整个问卷
- 基于用户回答的具体内容追问细节
- 不要让用户提供所有答案 —— 基于你的经验给出推荐方案，让用户选择/微调

## 结束条件
五个维度都清楚了 → 调用 sop_write 固化
```

### 0.5 Manager 回复用户（第 1 轮追问）

Manager LLM 读完 SKILL，调 `SendToHumanTool(type="info", content=...)`：

> 好的，我们一起把"小功能开发"的 SOP 定清楚。先问两个基础问题：
>
> 1. 你期望的团队构成是 Manager + PM + RD + QA 四个角色吗？还是希望再简化（比如合并 PM 和 Manager）？
> 2. 小功能的典型规模——是单接口修改，还是包含前后端的完整小应用？

`SendToHumanTool` 内部：
- 把卡片内容追加一条到 `shared/logs/l1_human/manager_to_human.jsonl`（L1 日志）
- 发飞书卡片（不触发 checkpoint_id，因为这是 `info` 类型不需要用户正式确认）
- 返回 ok

**关键点**：这是纯对话阶段，不写 `checkpoints.jsonl`（那是给正式 checkpoint 用的）。

### 0.6 用户回复 + Manager 多轮追问

用户："四个角色都要，小功能指 CRUD 级别的完整前后端小站。"

入站路径同 0.2。Manager Agent 再次 kickoff：
- `backstory` 注入时会自动加上本 session 的最近历史消息（22 课 `@before_llm_call` hook 做的）
- Manager 看到用户回答，继续追问维度 3-5

典型对话 3-5 轮后，五个维度都明确。

### 0.7 触发 sop_write（任务型）固化成 SKILL.md

Manager LLM 判断"维度全了"（依据 SKILL.md 里的"结束条件"），调 `load_skill("sop_write")`：

**sop_write SKILL.md**（任务型 → 启动 Sub-Crew 在沙盒内）：

```markdown
---
name: sop_write
type: task
---

# 固化 SOP 为 SKILL.md

## 输入
- content 里由 Manager 提供 5 个维度的对话摘要

## 步骤
1. 按 SOP 模板生成 SKILL.md 内容（见下）
2. 在沙盒内写到 /workspace/manager/skills/sop_{name}/SKILL.md
3. 写完 cat 一遍让 Manager 核对
4. 返回文件路径

## SOP 模板
---
name: sop_{name}
type: reference
---

# SOP: {项目类型}

## 概述
{一句话描述}

## 阶段清单
| 阶段 | Owner | 产物 | 评审方 | Checkpoint |
|------|-------|------|-------|-----------|
| ... | ... | ... | ... | ... |

## 详细步骤
### 阶段 1: xxx
- 目标: ...
- 所需 skill: ...
- 产物位置: ...
- 完成标志: ...
- 失败处理: ...

...

## 重要约束
- ...
```

沙盒在 `/workspace/manager/skills/sop_feature_dev/` 新建目录，写入 SKILL.md。

### 0.8 Manager 向用户确认 SOP

Manager 调 `SendToHumanTool(type="checkpoint_request", ...)`：

- `checkpoints.jsonl` 追加一条 `ck-sop-001`，记录 pending
- 飞书卡片展示 SOP 摘要 + "确认 / 修改" 按钮

用户回复"确认" → `recv_human_response` 匹配到 `ck-sop-001` → 写 `team:manager` 邮箱 → notify_role → Manager 醒来读到 approved → SOP 共创完成。

**此时 workspace 状态**：
```
workspace/manager/skills/sop_feature_dev/SKILL.md  ← 新建，内容为固化的 SOP
```

**后续所有"小功能开发"项目都用这同一个 SKILL**。共创只做一次。

## 阶段 2 · 需求澄清与需求文档

### 2.1 Manager 加载 requirements_discovery（参考型）

`requirements_discovery SKILL.md` 核心内容（27 课的四维发问框架）：

```markdown
---
name: requirements_discovery
type: reference
---

# 需求澄清发问框架

对新需求按四维追问：
- **目标**：这个任务成功的标准是什么？做给谁用？
- **边界**：必须做的是哪些？明确不做的是哪些？
- **约束**：时间、资源、技术栈、部署环境有什么限制？
- **风险**：哪些决策需要提前确认？（命名、存储、认证、多端?）

每次追问 1-2 个问题，基于用户前一轮回答推进。
不要一次抛出整个问卷，不要让用户写大段描述。

结束条件：四维都覆盖 + 无明显歧义 + 用户没有新补充。
结束后：load_skill("requirements_write") 把对话摘要写成正式需求文档。
```

### 2.2 Manager 和用户多轮对话

典型 3-5 轮：
- **R1 Manager**："做给你自己用还是多人用？需要登录吗？"
  用户："就我自己用，不用登录。"
- **R2 Manager**："数据本地存就行吗？（SQLite 文件）还是要云端？"
  用户："本地就行。"
- **R3 Manager**："宠物的字段：名字、品种、生日、头像——这些够吗？还要别的？"
  用户："够了。"
- **R4 Manager**："日记里的心情字段，happy/normal/sad 三档？还是自由文本？"
  用户："三档就行。"
- **R5 Manager**："前端要不要移动端适配？"
  用户："要。"

每一轮都通过 `SendToHumanTool(type="info", ...)` 和 `recv_human_response` → `team:manager` 邮箱实现。Manager Agent 每次被唤醒，上次对话作为 history 自动注入 backstory。

Manager 判断四维覆盖完，调 `load_skill("requirements_write")`。

### 3.1 Manager 读 checkpoint_response

Manager Agent kickoff：

### 3.2 sop_feature_dev SKILL.md（关键文档）

```markdown
---
name: sop_feature_dev
type: reference
---

# SOP: 小功能开发

## 阶段清单
| # | 阶段 | Owner | 产物 | 评审方 | 用户 Checkpoint |
|---|------|-------|------|-------|----------------|
| 1 | 需求澄清 | Manager | needs/requirements.md | —— | ✅ 确认需求 |
| 2 | 产品设计 | PM | design/product_spec.md | —— | —— |
| 3 | 产品文档评审 | Manager 协调 | design/reviews/*.md | RD + QA | —— |
| 4 | 产品文档 checkpoint | Manager | —— | —— | ✅ 确认设计 |
| 5 | 技术方案 | RD | tech/tech_design.md | —— | —— |
| 6 | 技术方案评审 | Manager 协调 | tech/reviews/*.md | PM + QA | —— |
| 7 | 代码实现 + 单测 | RD | code/, code 内自带单测 | —— | —— |
| 8 | 代码走查 | Manager 协调 | code/reviews/*.md | QA | —— |
| 9 | 测试设计 | QA | qa/test_plan.md | —— | —— |
| 10 | 测试设计评审 | Manager 协调 | qa/reviews/*.md | RD | —— |
| 11 | 测试执行 | QA | qa/test_report.md | —— | —— |
| 12 | 交付 | Manager | —— | —— | ✅ 确认交付 |
| 13 | 复盘进化 | Manager | proposals/*.md | —— | ✅ approve 提案 |

## 阶段推进规则

每次你（Manager）被唤醒，都要：
1. **先 read_sop_progress(project_id) 读 state.json** —— 知道当前在哪个阶段、pending 什么
2. **再 read_inbox()** —— 看新邮件
3. 根据 state + 邮件内容决定下一步动作

## 阶段详细 SOP

### 阶段 2 推进：发 PM 任务
- 调 send_mail(to=pm, type=task_assign, subject="产品设计", 
  content="读 needs/requirements.md 写 design/product_spec.md。按 PM 的 product_design SKILL 走。")
- 调 update_sop_progress: stage=2, status=in_progress, assignee=pm
- 当收到 pm 的 task_done：进入阶段 3

### 阶段 3 推进：发起团队评审
- 调 send_mail(to=rd, type=review_request, subject="评审产品文档", 
  content="读 design/product_spec.md, 以 RD 视角评审（可实现性、技术风险）。
  写评审意见到 design/reviews/rd_review.md。通过/不通过 + 具体意见。")
- 同步对 qa 发 review_request
- update_sop_progress: stage=3, pending_reviewers=[rd, qa]
- 收到两份评审后汇总：
  - 都 pass → 进入阶段 4
  - 任一 fail → 整理意见发回 pm 修改，回到阶段 2

### 阶段 4 推进：产品文档 checkpoint
- 调 send_to_human(type=checkpoint_request, summary=产品设计摘要)
- 等 checkpoint_response：approved → 阶段 5；rejected → 根据意见决定回谁

### 阶段 5 推进：RD 技术方案
- 调 send_mail(to=rd, type=task_assign, subject="技术方案", 
  content="读 needs + design, 写 tech/tech_design.md. 按 RD 的 tech_design SKILL 走")
- 收到 task_done → 阶段 6

### 阶段 6 推进：技术方案评审
- 发 review_request 给 pm + qa
- 收到两份意见汇总：pass → 阶段 7；fail → 回 rd

### 阶段 7 推进：代码实现
- send_mail(to=rd, type=task_assign, subject="代码实现", 
  content="按 tech_design 实现,code/ 目录, 带 pytest, coverage>=80%。按 code_impl SKILL 走。")
- 收到 task_done → 阶段 8

### 阶段 8 推进：代码走查
- send_mail(to=qa, type=review_request, subject="代码走查",
  content="从测试视角走查 code/, 输出 code/reviews/qa_code_review.md")
- 收到意见: pass → 阶段 9; 有阻塞问题 → 回 rd

### 阶段 9 推进：测试设计
- send_mail(to=qa, type=task_assign, subject="测试设计",
  content="按 test_design SKILL 走, 输出 qa/test_plan.md")
- task_done → 阶段 10

### 阶段 10 推进：测试设计评审
- send_mail(to=rd, type=review_request, subject="评审测试设计",
  content="从技术实现角度评审 qa/test_plan.md 是否覆盖关键路径")
- pass → 阶段 11; fail → 回 qa
```

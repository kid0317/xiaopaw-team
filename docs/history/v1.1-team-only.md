## 1. 项目定位

xiaopaw-team 是模块四的实战收官项目。它**不是**从头造一个新系统，而是一次**工程装配练习**——把 25-28 课已经讲过的机制装配起来，端到端做完一个 toC 项目，并完成一轮自我进化。

**与 22 课 xiaopaw-with-memory 的关系**：

| 维度 | 22 课 | 29 课 xiaopaw-team |
|------|------|------|
| Agent 数量 | 1 个 Main Agent | 4 个独立 Main Agent（Manager / PM / RD / QA） |
| 人机通道 | 飞书 ↔ Main Agent | 飞书 ↔ **只和 Manager 通信**（唯一架构保证） |
| 团队内协作 | 无 | 文件邮箱（26 课三态）+ 事件流（本课新增） |
| Agent 触发源 | 飞书消息 | 飞书消息（只进 Manager）/ 邮件唤醒 / cron 兜底 |
| 业务流程 | 单次对话 | 端到端项目 + 复盘 + 进化 |

**课程价值**：验证"25-28 的机制同时在线时是否真能协作做完项目"。装配过程中需要补的接缝（唤醒、单沙盒、单一接口、SOP 作为 skill、事件流推断 state）就是本课工程增量。

## 2. 核心设计原则

### 2.1 单一接口原则（27 课，硬约束）

**人永远只和 Manager 沟通**。三重架构保证：

- 飞书入站：FeishuListener 只把消息路由给 Manager agent_fn
- 飞书出站：`SendToHumanTool` 只注入 Manager 的 Crew 构造器
- PM / RD / QA 的 Crew 构造器**不接受** `feishu_sender` 参数——Python 对象层面传不进去，LLM 在 SKILL 里调 `send_to_human` 会报 "Tool not found"

任何"团队内角色想跟用户说话"的场景，只能发邮件给 Manager，由 Manager 翻译成飞书卡片。

### 2.2 零 Python 编排原则

**流程推进逻辑 100% 写在 SKILL.md 里，Python 代码不做任何业务流程决策**。

代码只做 5 件事：
1. **启动**：main.py 构造 4 个 MemoryAwareCrew（对应 4 个 workspace）
2. **路由**：Runner 根据 InboundMessage.routing_key 分派到对应 agent_fn
3. **工具**：Python 函数作为 CrewAI Tool 暴露（发邮件、读状态、写共享文件等）
4. **沙盒**：调用 AIO-Sandbox 执行 task 型 Skill 的脚本
5. **唤醒**：CronService 把时间/事件转成 InboundMessage

**严禁的反模式**：
- ❌ `if stage == "product_design" then next = "product_design_review"`（流程编排不能在 Python）
- ❌ `if reviewer_count == 2 then advance()`（评审汇总不能在 Python 计数）
- ❌ 预定义的 stages 列表（state 结构不能对某个 SOP 有知识）
- ❌ `if skill_name.startswith("sop_"):`（不能靠命名约定识别 skill 类型）

### 2.3 Agent = Crew + SkillLoaderTool + 四层框架（25 课）

每个角色：1 个 `MemoryAwareCrew @CrewBase`（22 课复用）+ 1 个 `SkillLoaderTool` + 若干协作工具。`backstory` 由 `build_bootstrap_prompt(workspace/{role}/)` 从四层文件拼接：soul / agent / memory / user。

**LLM 决定加载哪个 skill** 靠两个信号：
- SKILL.md frontmatter 的 `description`（SkillLoader 汇总成工具描述）——**主要信号**
- agent.md 里的元规则（"每次 kickoff 先 read_state → read_inbox 两步"）——**次要信号**

agent.md **禁止**写"收到 type=X 的消息加载 skill Y"这种硬路由表，路由全靠 description 特征词。

### 2.4 状态共享 vs 任务触发（26 课）

- **共享工作区**（文件持久化状态）：需求、设计、代码、测试报告
- **文件邮箱**（JSON 结构化事件）：task_assign / task_done / review_done
- 邮件只传路径引用，绝不复制文档正文

### 2.5 SOP 驱动 + 事件流推断 state（v1.0 关键）

**没有**预定义的 stages 列表。项目状态的真相来自两处：
- **append-only 事件流** `shared/projects/{pid}/events.jsonl`：每次 Manager 推进都追加一条事件
- **文件系统痕迹**（produce/design/code/reviews/qa/ 下有哪些文件）

Manager 每次 kickoff 第一步是 `read_project_state(project_id)`——这是一个 task 型 skill（不是工具），脚本读 events.jsonl 尾部 + ls 关键目录 → 产出"当前状态摘要"给 Manager LLM。由 LLM 根据 sop_feature_dev SKILL 内容决定下一步，**不是 Python 代码决定**。

### 2.6 进化闭环（28 课）

每次项目跑完触发一次复盘。提案严格分档 HITL（档 1 auto / 档 2 user approve / 档 3 手工不演示）。本课演示到"列出提案 + apply 到 workspace 文件 + git commit"为止。

### 2.7 不自建调度，复用 xiaopaw CronService

每个角色启动时注册 1 个 `every 30s + jitter 5s` 兜底 cron；发邮件由 `SendMailTool` 内部注册 `at=now+1s` one-shot；三类触发源（飞书、邮件唤醒、cron 兜底）都构造 `InboundMessage` 进 Runner，对 Agent LLM 统一。

## 3. Demo 项目：小爪子日记

### 3.1 产品描述

toC 宠物日记单页网站。选它的理由：呼应 xiaopaw 品牌、CRUD 量足以让 RD 写分层代码、前后端都真实、有清晰可测性。

### 3.2 功能清单（MVP）

**实体：**
- `Pet`: id, name, species, birthday, avatar_url, created_at
- `Diary`: id, pet_id, date, content, mood(happy/normal/sad), photos[], created_at

**后端 REST API：**
- Pets：`POST/GET/PUT/DELETE /api/pets[/id]`
- Diaries：`POST/GET /api/pets/{id}/diaries`、`DELETE /api/diaries/{id}`

**前端：**
- `/`：宠物列表卡片 + 添加按钮
- `/pets/{id}`：详情 + 日记时间线

### 3.3 技术栈（Skill 推荐 + 硬约束）

不在架构文档里写死技术栈。`rd/skills/tech_design/SKILL.md` 里定义：

**推荐栈（非强制）**：FastAPI + SQLAlchemy 2.0 + SQLite / 原生 HTML+JS+CSS / pytest + httpx
**硬约束**：带 pytest 且全 PASS / coverage 达到 `team_standards.pytest_coverage_min`（默认 80%）/ 后端分层 / 前端无构建工具 / **禁止启动长期进程（uvicorn、daemon）**，所有 HTTP 测试用 `httpx.Client(app=fastapi_app)` 直连

### 3.4 范围边界

不做：用户系统 / 图片上传对象存储 / 前端构建工具链 / 部署 / CI/CD。

### 4.2 关键决策

- **单进程**：4 Agent 在同一 Python 进程（一个 routing_key 一个 agent_fn）
- **单沙盒**：全团队共享 1 个 AIO-Sandbox，目录前缀区分 Owner，工具层校验权限
- **Manager 并发保护**：agent_fn 外包 `asyncio.Lock`——同时接到飞书消息和 cron wake 时串行处理（避免两个 Manager 实例同时写 events.jsonl）
- **session_id 策略**：
  - `p2p:{open_id}` session_id = 每个用户独立（pgvector 隔离用户历史对话）
  - `team:{role}` session_id = 每角色一个固定 id（跨 project 共享，是"团队协作本能"的积累载体）

### 4.3 为什么保留 p2p routing_key 而不统一 team:manager

v0.3 曾建议把飞书入站也改成 team:manager，但这会导致：
- 原 `p2p:{open_id}` 的 session 含义（每用户独立会话历史）丢失
- pgvector 的用户视角记忆和团队视角记忆混入同一 session，检索时噪声大

v1.0 选择：**routing_key 按原语义保留**，Manager 并发靠 `asyncio.Lock` 解决（更便宜、更干净）。

## 6. 沙盒与执行隔离

### 6.1 为什么单沙盒

生产场景应该每角色独立沙盒（隔离、可伸缩），但本课演示机（一般机器）撑不住 4 个容器。v1.0 选择单沙盒 + 目录隔离 + 工具层权限校验。

### 6.2 目录隔离

```
/workspace/
├── manager/ pm/ rd/ qa/           ← 角色私有目录（Sub-Crew 启动时 Bootstrap 注入但不独占）
└── shared/projects/{pid}/
    ├── needs/    ← Manager 写
    ├── design/   ← PM 写
    ├── tech/     ← RD 写
    ├── code/     ← RD 写
    ├── qa/       ← QA 写
    └── reviews/{stage}/{reviewer}_{topic}.md  ← 任意角色写自己的评审（命名规则见 6.3）
```

### 6.3 权限表（工具层强制）

`tools/workspace.py` 内 `OWNER_BY_PREFIX`：

```python
OWNER_BY_PREFIX = {
    "needs/":   {"manager"},
    "design/":  {"pm"},
    "tech/":    {"rd"},
    "code/":    {"rd"},
    "qa/":      {"qa"},
    # reviews/ 特殊：任何角色可写，但只能写 reviews/{any}/{self_role}_*.md
}

def check_write(role, rel_path):
    if rel_path.startswith("reviews/"):
        # reviews/{stage}/{reviewer}_xxx.md  —— reviewer 必须是 role 自己
        parts = rel_path.split("/")
        if len(parts) >= 3 and parts[2].startswith(f"{role}_"):
            return
        raise PermissionError(f"{role} can only write reviews/*/{role}_*.md")
    owners = OWNER_BY_PREFIX.get(prefix_of(rel_path))
    if not owners or role not in owners:
        raise PermissionError(f"{role} cannot write {rel_path}")
```

评审意见放独立目录，不嵌套产物目录，避免 prefix 套娃。

**文件命名规则**（工具层强制）：
- 路径必须匹配正则 `^reviews/[a-z_]+/([a-z]+)_([a-z_]+)\.md$`
- 第 1 个捕获组（文件名下划线前部分）= 当前调用角色（`manager` / `pm` / `rd` / `qa`），不匹配直接拒绝
- 第 2 个捕获组 = topic（如 `review`, `quick_check`, `final_signoff`），自由命名但只能小写字母和下划线

示例：`reviews/product_design/rd_review.md` ✅；`reviews/product_design/qa_acceptance_check.md` ✅；`reviews/product_design/manager.md` ❌（没有 topic 后缀）；`reviews/code/pm_review.md` ❌（PM 角色不做代码评审，按 sop_feature_dev 规则不应写）

### 6.4 禁止启动长期进程

所有 HTTP 接口测试**必须**用 `httpx.Client(app=fastapi_app)` 直连（FastAPI 的 TestClient 模式），不允许 `uvicorn ... &`、`python -m http.server &` 等启动后台进程。

理由：
- RD 和 QA 共享同一沙盒，同时跑 uvicorn 会占用同一端口
- TestClient 模式更快（无 HTTP 开销），pytest 用例间也干净（每 test 独立 client）
- SQLite 每个测试用独立 tmpfile（`tempfile.NamedTemporaryFile()`），避免 RD 单测库和 QA 集成测试库冲突

写在 `code_impl` 和 `test_run` 两个 SKILL 的硬约束里。

## 7. 唤醒机制

### 7.1 三条唤醒路径

| 来源 | 触发方式 | 目的 |
|------|---------|------|
| 飞书用户消息 | FeishuListener → Runner (`p2p:{open_id}`) | 用户发起新任务或回复 checkpoint |
| SendMailTool 内部 | 注册 `at=now+1s` one-shot cron | 即时唤醒收件人（主路径） |
| 每角色 every 30s | CronService `every 30s + jitter 5s` | 兜底：异常后恢复、手工干预后补偿 |

对目标 Agent 来说，三者都是 Runner 送来的 `InboundMessage`，处理逻辑统一。

### 7.2 InboundMessage 扩展

```python
@dataclass
class InboundMessage:
    routing_key: str
    text: str
    meta: dict    # v1.0 新增 wake_reason
    # wake_reason ∈ {"feishu", "new_mail", "heartbeat"}
    # agent LLM 读到 heartbeat 且 inbox 空时可以直接返回不调任何工具
```

### 7.3 cron wake 去重

Runner.dispatch 对 `routing_key=team:{role}` 且 `meta.wake_reason ∈ {new_mail, heartbeat}` 的消息，做入队去重：

```python
async def dispatch(self, inbound):
    if inbound.routing_key.startswith("team:") and inbound.meta.get("wake_reason") in {"new_mail", "heartbeat"}:
        queue = self._queues.get(inbound.routing_key)
        if queue:
            # 检查队列里是否已有相同 wake_reason 的 pending
            for pending in list(queue._queue):
                if pending.meta.get("wake_reason") in {"new_mail", "heartbeat"}:
                    return   # 丢弃本次
    await self._enqueue(inbound)
```

防止 "at one-shot 和 every 30s 同时触发 → Agent 多跑一次空 kickoff"。

### 7.4 SendMailTool 联动

```python
def send_mail(to, type_, subject, content, project_id, from_):
    # 1. 写邮箱（filelock 三态）
    mailbox_ops.send_mail(...)
    # 2. 即时唤醒
    cron_service.register_job({
        "id": f"wake-once-{uuid4()}",
        "schedule": {"kind": "at", "timestamp_ms": now_ms() + 1000},
        "payload": {
            "routing_key": f"team:{to}",
            "text": f"[wake] 新邮件到达",
            "meta": {"wake_reason": "new_mail"},
        },
    })
    # 3. task_callback 会自动写 L2（kickoff 完成时），这里不动
```

### 7.5 Manager 并发保护

Manager 可能被两个 routing_key 触发（`p2p:{open_id}` 飞书，`team:manager` cron wake）。外包 asyncio.Lock：

```python
async def manager_agent_fn(text, history, session_id, routing_key, root_id, verbose):
    async with MANAGER_LOCK:
        return await _original_manager_agent_fn(...)
```

Runner 的 per-routing_key 串行队列仍然有效，但跨 routing_key 的 Manager 调用通过这个锁串行化。副作用：用户正在对话时，cron wake 会短暂等待（最多几秒），对用户体验几乎无感。

## 9. 协作机制：邮箱 + 共享区 + 事件流

### 9.1 文件邮箱（26 课原样）

- FileLock 保护 `读→追加→写`
- 三态 `unread → in_progress → done`
- `reset_stale` 作为未来扩展（本课单进程不启用）

位置：`shared/projects/{pid}/mailboxes/{role}.json`

### 9.2 共享工作区

见 §6.2 目录结构。**不再有 human.json**——Manager ↔ 用户通过飞书通道 + `events.jsonl` 记录审计。

### 9.3 事件流 events.jsonl（v1.0 关键）

**append-only 事件记录**。Manager 每次推进都追加一条事件（行级 JSON），**没有预定义 stages 列表**。

结构见附录 A。

**写入规则**：
- 仅 Manager 有权写 events.jsonl（工具层强制）
- FileLock 保护 append（防 agent_fn 并发写）
- 每条事件至少含 `timestamp / stage_id / action / actor / payload`

**读取规则**：
- `read_project_state` skill 的脚本：tail events.jsonl + ls 关键目录 → 生成状态摘要
- Manager LLM 读摘要自己推断"下一步应该做什么"（SOP SKILL 指引）

**优点**：
- 没有 schema 硬编码（stages 列表随 SOP 事件自然生长）
- 演化友好：新 SOP 只需新 event type，不改代码
- 审计完整：append-only 不丢历史

**缺点**：
- 摘要生成有一定 LLM 推理成本（但 `read_project_state` 脚本可预处理，把事件流压缩成 50-100 行摘要）

### 9.3.1 鲁棒性：mark_done 与 append_event 的顺序约束

**关键不变式**：Manager 处理团队邮件（尤其是 checkpoint_response）时，必须严格按顺序：
```
1. read_inbox → 取到消息（自动标 in_progress）
2. 调 append_event（记录已处理的事件，如 checkpoint_approved）
3. 调 mark_done（标记消息已处理）
```

**不能**先 mark_done 再 append_event——如果在中间崩溃，事件丢失而 mailbox 已记完成，下次恢复时 read_project_state **无法从 events.jsonl 看到这个 checkpoint_response 已被处理**，Manager 会误以为"用户还没回"，再次给用户发卡片。

### 9.3.2 read_project_state 的自愈检查

脚本在生成摘要前做一步**交叉校验**：
```
对每条 status=done 的 checkpoint_response 邮件：
  查 events.jsonl 是否有对应 checkpoint_approved/rejected 事件
  若缺失 → 追加一条 "recovered_checkpoint_response" 事件（来自 mailbox 推断）+ 告警日志
```

这样即使发生崩溃半写，下一次 Manager kickoff 就能自动补齐事件流，不依赖人工介入。

### 9.3.3 tail 而非全读

events.jsonl 长期项目可达到几百条。脚本默认只 tail 最后 100 条事件（`team_standards.event_tail_window`），超出窗口的老事件摘要另存 `shared/projects/{pid}/events_summary.json`（每完成一个主阶段就 flush 一次摘要，滚动归档老事件）。

### 9.4 与 checkpoints.jsonl 的关系

v0.3 引入过 `checkpoints.jsonl`（审计用户 checkpoint）。v1.0 **合并进 events.jsonl**——用户 checkpoint 也是一种 event（`action="checkpoint_requested"` / `"checkpoint_responded"`）。

## 10. 飞书桥与单一接口原则

### 10.1 入站：feishu_bridge.recv_human_response

```python
def recv_human_response(feishu_msg, routing_key):
    # 1. 检查最近 events.jsonl 有无 pending checkpoint
    pending_ck = find_pending_checkpoint_from_events(routing_key)
    if pending_ck:
        # 作为 checkpoint_response 写 manager.json
        send_mail(to="manager", from_="human",
                  type_="checkpoint_response",
                  project_id=pending_ck["project_id"],
                  content={"ck_id": pending_ck["id"],
                           "text": feishu_msg.text,
                           "decision": classify(feishu_msg.text)})
        # SendMail 内部已 at cron 唤醒 Manager
        return "consumed_as_checkpoint"
    # 常规飞书对话：Runner 按 p2p:{open_id} 路由到 Manager agent_fn，
    # 但 Runner 的 dispatch 本身不会触发 cron wake，为了避免用户
    # 发消息后等 30s heartbeat 才被处理，显式额外注册一个 at wake（noop 也无妨，去重生效）
    notify_role_at("manager", delay_sec=0)   # 飞书路径天然就是同步，这个 notify 只是兜底
    return None
```

**两类入站路径的统一保证**：
- 检测到 pending checkpoint → 走 mailbox（内嵌 at wake）
- 无 pending checkpoint → Runner 正常 p2p 路由（飞书 session 正常延续）+ 兜底 at notify

两条路径都不依赖 every 30s heartbeat 就能及时响应用户。

### 10.2 出站：SendToHumanTool

```python
def send_to_human(project_id, type_, title, summary, artifacts=None):
    # 1. 追加 events.jsonl: action=checkpoint_requested（或 info_sent / delivery_sent / evolution_sent）
    event_id = append_event(project_id, action=f"{type_}_sent", payload={...})
    # 2. 飞书卡片
    card = build_card(type_, title, summary, artifacts)
    msg_id = feishu_sender.send_card(user_open_id, card)
    # 3. L1 日志
    log_l1(action=f"{type_}_sent", project_id=project_id, content={title, summary})
    return event_id
```

卡片 type 枚举：`info` / `checkpoint_request` / `proposals_review` / `delivery` / `evolution_report` / `error_alert`

### 10.3 单一接口的架构保证

`build_role_tools(role, cfg)` 只给 `role == "manager"` 注入 `feishu_sender`。其他角色的 SkillLoaderTool 里根本没注册 `send_to_human`，LLM 调用会报 "Tool not found"。架构级保证，不靠 prompt 约束。

- 它需要对 `workspace/` 目录做 `git commit`，但 AIO-Sandbox 内装 git 并操作挂载目录的 git 仓库会有权限/进程交叉问题

- 所以 ApplyProposalTool 作为 Manager 主 Agent 的一个 Tool（不是 task skill，不走 Sub-Crew/沙盒），在主进程直接 subprocess 调 `git add` + `git commit`
- 这也意味着 `workspace/` 必须是一个**独立的 git 仓库**（见 §23.3 初始化步骤）

## 13. SOP 作为 reference skill

### 13.1 SkillLoader 如何找到"哪些 skill 是 SOP"

**不靠命名约定**（`sop_` 前缀是风格建议，不是架构约束）。靠 **frontmatter `kind` 字段**：

```yaml
---
name: sop_feature_dev
description: 小功能开发主流程。收到新开发任务后加载。
type: reference
kind: sop          # ← 本课新增
version: 1.0.0
---
```

`sop_selector` skill 调用脚本 `list_sops.py`，扫描 Manager 的 skills 目录（由 main.py 传入 `workspace/manager/skills/`）：

```python
def list_sops(skills_dir):
    sops = []
    for skill_dir in Path(skills_dir).iterdir():
        skill_md = skill_dir / "SKILL.md"
        if not skill_md.exists(): continue
        fm = parse_frontmatter(skill_md)
        if fm.get("kind") == "sop":
            sops.append({"name": fm["name"], "description": fm["description"]})
    return sops
```

**SkillLoader 的三项扩展**（见 §24.2 详细改动）：
1. 从"manifest 驱动"切换到"按角色扫描 workspace/{role}/skills/ 目录"——或保留 manifest 改造为 4 份（每角色一份 `workspace/{role}/skills/load_skills.yaml`），二选一
2. frontmatter 增加 `kind` / `owner_role` 字段的识别
3. description 支持 `{team_standards.X}` 变量替换

本课推荐方案：**4 份 manifest**（每个角色 workspace 下一份），减少扫描开销、保持 22 课 manifest 风格的一致性。

```markdown
# SOP: 小功能开发

你是 Manager。收到用户新开发需求后，按以下推进规则执行。

## 阶段清单（6 主阶段）

| # | 阶段 | Owner | 产物 | 用户 Checkpoint | 可选团队评审 |
|---|------|-------|------|----------------|-------------|
| 1 | 需求澄清 | Manager | needs/requirements.md | ✅ | —— |
| 2 | 产品设计 | PM | design/product_spec.md | ✅ | 可插入 RD+QA |
| 3 | 技术设计+实现+单测 | RD | tech/tech_design.md, code/ | —— | 可插入（技术方案）PM+QA |
| 4 | 代码走查+测试+执行 | QA | qa/test_plan.md, qa/test_report.md | —— | 代码走查 by QA 强制 |
| 5 | 交付 | Manager | —— | ✅ | —— |
| 6 | 复盘进化 | Manager | proposals/* | ✅（逐条 approve） | —— |

## 推进规则总则

每次被唤醒，先：
1. `load_skill("read_project_state")` → 拿到当前状态摘要
2. `read_inbox()` → 看邮件
3. 根据两者决定下一步，按下表处理：

| 入邮件 | 当前阶段 | 下一步 |
|--------|---------|--------|
| 无新邮件 | any | 看是否有挂起 checkpoint 未回复（几乎不会，因为 checkpoint 等待是异步的） |
| feishu 新任务 | 无 current project | create_project + 进阶段 1 |
| checkpoint_response(approved, ck=req) | 1 | 进阶段 2（见"阶段 2 推进"） |
| checkpoint_response(approved, ck=design) | 2 | 进阶段 3 |
| checkpoint_response(approved, ck=deliver) | 5 | 进阶段 6 |
| pm 的 task_done | 2 | 验收 + 决定是否插评审 / 直接 checkpoint |
| review_done (pass 或汇总后 pass) | 任意评审中 | 进下一阶段 |
| review_done (need_changes) | 任意评审中 | 整理意见发回产出角色 |
| rd 的 task_done(tech_design) | 3 | 决定是否插入评审 |
| rd 的 task_done(code) | 3 | 发 QA 代码走查（阶段 4） |
| qa 的 task_done(test_report, 无阻塞) | 4 | 进阶段 5 |
| qa 的 task_done(test_report, 有阻塞缺陷) | 4 | 发回 rd 修复 |
| error_alert | any | 发 error_alert 到用户，等人工判断 |

## 阶段详细规则

### 阶段 1 推进（v1.1：拆 guide + write）
- 生成 project_id（LLM 决定），调 CreateProjectTool 初始化工作区
- append_event(action="project_created")
- load_skill("requirements_guide") ← reference，获得四维发问框架
- 基于框架调 SendToHumanTool 发问；等用户回复后继续（新一次 kickoff 看 history 判断进度）
- 四维覆盖完 → load_skill("requirements_write", task_context={project_id, 摘要}) ← task
- Sub-Crew 返回路径后：append_event(action="requirements_drafted")
- send_to_human(checkpoint_request, summary=需求摘要)

### 阶段 2 推进
- send_mail(to=pm, type=task_assign, subject="产品设计", content="读 needs/...写 design/...")
- append_event(action="assigned", to=pm, stage=product_design)
- 收到 pm task_done 后：
  - 读 design/product_spec.md 基础抽检（文件存在、非空、含模板必要节）
  - `load_skill("check_review_criteria")` → 脚本机械数出实体数、API 数并返回数字
  - 比对 team_standards.yaml 阈值（by LLM）：实体 >= min_entities 且 API >= min_apis → 插入 RD+QA 评审
    （脚本只"数"、LLM 只"比对与决策"，消除 LLM 数错 + 脚本塞业务逻辑的二难）
  - 插入评审：见"评审子流程"
  - 不插入：直接 send_to_human(checkpoint_request, summary=设计摘要)
  - append_event(action="reviewed" 或 "skipped_review")

### 阶段 3 推进
- send_mail(to=rd, type=task_assign, subject="技术设计 + 代码实现",
  content="按 tech_design SKILL 和 code_impl SKILL, 
  先写 tech/tech_design.md, 自查通过后实现 code/, pytest 全 PASS")
- 可选插入技术评审：收到 RD 发的 tech_done 中间回执时（如有），发 review_request 给 pm + qa
- 收到 rd task_done(code) 后：直接进阶段 4（代码走查是阶段 4 的一部分）
- append_event(action="rd_delivered", artifacts=[code/, tech/])

### 阶段 4 推进
- 4a 代码走查（强制）：send_mail(to=qa, type=review_request, subject="代码走查")
  qa 返回 pass + 可能的非阻塞意见 → 进 4b
  qa 返回 need_changes(阻塞) → 发回 rd 修
- 4b 测试设计：send_mail(to=qa, type=task_assign, subject="测试设计")
  可选插入 rd 评审 test_plan（按 team_standards.review_test_plan_threshold 决定）
- 4c 测试执行：send_mail(to=qa, type=task_assign, subject="测试执行")
  无阻塞缺陷 → 进阶段 5
  有阻塞 → 发回 rd 修 + qa 回归
- append_event(每个小步都 append)

### 阶段 5 推进
- send_to_human(type=delivery, summary=最终交付摘要, artifacts=[code/README.md, qa/test_report.md])
- 等 approved → 进阶段 6
- append_event(action="delivered")

### 阶段 6 推进（v1.1 修订：区分 team_retro / review_proposals / apply_proposal 三步）
- **6a 团队复盘触发**：`load_skill("team_retrospective")` task → Sub-Crew 读 L1/L2 聚合，返回瓶颈摘要 + "建议对 [role1, role2, ...] 发 retro_trigger"
- 主 Agent 按建议对每个角色 `send_mail(type=retro_trigger)`
- append_event(action="retro_triggered", to=[...])
- **6b 收各角色 retro_proposal**（每个角色发一条，主 Agent 被邮件唤醒多次）
  - 每次被唤醒 → read_inbox 看到 retro_proposal
  - `load_skill("review_proposals")` reference → 获得预审 checklist + 汇总规则
  - 主 Agent 按 checklist 对本条 proposal 做合规检查
  - 不合格 → append_event(action="proposal_rejected", reviewer=..., reason=...), mark_done（不打扰用户）
  - 合格 → append_event(action="proposal_received", reviewer=..., proposal_id=...)
  - 查 events.jsonl 看是否收齐：`expected = retro_triggered.to.length`，`received = proposal_received.count`
    - 未收齐 → mark_done，等下一条
    - 收齐 → 进 6c
- **6c 汇总请用户 approve**：send_to_human(type=proposals_review, summary=提案列表, artifacts=[N 份提案 md])
- **6d 用户 approve**（一次性全部 approve 或逐条）：
  - 收到 checkpoint_response(type=proposals_approve, approved_ids=[...])
  - `load_skill("handle_checkpoint_reply")` 或直接进入：
  - 对每个 approved id 调 `ApplyProposalTool(proposal_id)` ← Tool 不是 skill
  - append_event(action="proposal_applied", proposal_id=..., commit=...)
  - 全部应用完 → send_to_human(type=evolution_report, 附 git diff 摘要)
  - append_event(action="evolved")

## 评审子流程（被阶段 2 / 阶段 3 / 阶段 4b 调用）
- 发 review_request 给两方 / 一方
- 等两份 review_done
- 汇总：两方都 pass → 通过；有 need_changes → 发回产出角色修
- **轮次写进 subject**：发回修改时 subject 必须含"(第 N 轮)"（如"产品设计修订 (第 2 轮)"），让产出角色也能从邮件感知轮数
- 循环上限 `team_standards.review_max_rounds`（默认 3），超过 → send_to_human(error_alert)

## 硬约束
- 每次动作前 `read_project_state`，每次动作后 `append_event`
- 你从不写代码 / 设计 / 测试
- 你从不让 PM/RD/QA 联系用户
- 评审循环不可超上限
- 所有 checkpoint 必须等用户回复后才能推进
```

**关键设计**：14 阶段压成 6 主阶段，评审只有"代码走查"强制；其他评审为"有条件插入"，由 sop_feature_dev SKILL 自己根据规模判据决定。这样既不削弱课程"交叉评审"的演示意图，又不把 CRUD 小 demo 拖到 14 阶段。

阈值判据（是否插入评审）写在 `team_standards.yaml`，不写死在 skill 文本：

```yaml
review_insert_product_design:
  min_entities: 3
  min_apis: 7
review_insert_tech_design:
  min_lines_estimated: 500
review_test_plan_threshold:
  min_cases: 10
review_max_rounds: 3
```

### 13.3 Skill 类型 vs 主 Agent 工具的能力边界（v1.1 新增）

这是 29 课 skill 设计的核心原则。三种"动作载体"各有能力边界，不能混用：

| 载体 | 能做 | 不能做 | 典型用途 |
|------|------|--------|---------|
| **reference skill** | 把 SKILL.md 正文塞回主 Agent 上下文，主 Agent 基于指令推理 | 不能操作文件；不能跑代码；不能对用户发消息；不能多轮等回复 | **对话框架** / 评审 checklist / 主流程 SOP / 决策树 |
| **task skill** | 启动独立 Sub-Crew + AIO-Sandbox；可读写文件、跑 bash、跑 python | **中途不能等用户回复**（一次 kickoff 就退出）；看不到主 Agent 的飞书 session；看不到主 Agent 已有对话历史（除非 inputs 显式传入） | **一次性生成长文档** / 跑 pytest / 执行 git commit（但 git 走宿主机 Tool 不是 task skill）/ 机械数据处理 |
| **主 Agent 直接 Tool** | 原子动作：发邮件、收邮箱、发飞书卡片、读/写 markdown、追加 event | 复杂逻辑（如"按模板生成 2000 字设计文档"）不适合，会污染主 Agent 上下文 | 团队协作基础设施：SendMail/ReadInbox/MarkDone/ReadShared/WriteShared/SendToHuman/AppendEvent/CreateProject/ApplyProposal |

### 13.3.1 为什么 29 课主 Agent 必须有直接 Tool

22 课原教义："Main Agent 只有 SkillLoaderTool"——当时场景是"用户 → 主 Agent → 调 skill 完成任务 → 回复用户"的单轮流程。

29 课场景不同：主 Agent（尤其 Manager）要维持项目状态、发多种邮件、跟多个角色协作、主动推送飞书。如果全走 skill：
- 发一封邮件要启动一次 task skill（Sub-Crew 启动开销 ~ 几秒 + 额外 LLM 调用），每次 kickoff 可能发 2-5 封邮件 → 开销爆炸
- 多轮对话中主 Agent 要"基于对话框架发下一个问题"，SkillLoader 不提供"发问题"的原子能力
- 每次 read_inbox 都要启动 Sub-Crew 没有意义

所以 29 课把**"团队协作基础设施"下沉到主 Agent 的直接 Tool**：发邮件、收邮箱、发飞书、读写共享文件、追加事件。这是对 22 课单 Agent 场景的合理扩展，**SkillLoaderTool 仍然是"领域能力"的扩展入口**（对话框架、SOP、复杂文件生成、复盘），工具和 skill 职责清晰分层。

### 13.3.2 "对话 + 写入"类 skill 的标准拆分模板

凡是**"需要多轮对话 + 对话结束后写长文档"**的场景，都按此模板拆两个 skill：

```
xxx_guide  (reference)
├─ 提供对话框架、发问策略、结束条件判据
├─ 主 Agent 加载后：反复 SendToHuman → ReadInbox → 判断是否结束
└─ 结束时 → 调 xxx_write

xxx_write  (task)
├─ 输入：{对话摘要 / 关键参数}（主 Agent 从自己的对话历史摘要传入）
├─ Sub-Crew 在沙盒内按模板生成长文档
└─ 返回：文件路径
```

本课按此模板拆分的 skill：
- `sop_cocreate_guide` + `sop_write`（SOP 共创）
- `requirements_guide` + `requirements_write`（需求澄清）

本课**不**按此拆分的"对话型"场景（特殊）：
- **需求修订**：用户驳回 checkpoint 后的修订不是新一轮澄清，是 Manager 直接 send_mail 给 PM 附修改意见，不需要 guide skill
- **handle_checkpoint_reply**：主 Agent 直接根据用户回复做分类决策，不需要对话框架（反而是"动作路由"性质），reference 型够用

### 13.3.3 SOP 是什么类型的 skill（用户追问）

**SOP 是 reference skill**，frontmatter 额外标 `kind: sop`：

- **不能是 task**：task 是"一次性 Sub-Crew 跑完返回"，而 SOP 是"主 Agent 持续参考的指令文档"——每次 Manager 被唤醒做决策前都需要看 SOP 的推进规则。语义上只能是 reference
- **不能是普通共享文件**：如果 SOP 只放在 `shared/sop/xxx.md` 让主 Agent 用 ReadShared 读取，SkillLoaderTool 的工具 description XML 里看不到 SOP 的存在，Manager LLM 不知道有哪些 SOP 可选，sop_selector 也无法枚举
- **reference + kind=sop 两全其美**：SkillLoader 枚举（可见）+ sop_selector 过滤（可选）+ reference 语义（持续参考）

注：sop_cocreate_guide 产出的**新 SOP 同样是 reference kind=sop 型**，sop_write 的 Sub-Crew 生成 `workspace/manager/skills/sop_{name}/SKILL.md` 时 frontmatter 必须带 `kind: sop`。

**新 SOP 如何被 SkillLoader 注册**（工程细节）：
- SkillLoader 启动时加载 manifest 构建工具 description，**运行时新增 skill 不会自动生效**
- v1.1 解法：`sop_write` 任务除了写 SKILL.md，还要调 `workspace/manager/skills/load_skills.yaml` 的 manifest 追加一条
- main.py 监听 manifest 文件 mtime 变化，触发 SkillLoader 重新 `_build_description()`（类似 22 课 CronService 的 `_load_store()` 热重载模式）
- 下次 Manager kickoff 时（因为是每次 kickoff 构造 agent_fn 里才创建 SkillLoaderTool 实例？实际是每次 kickoff 都是同一个 SkillLoaderTool —— 需要 SkillLoader 本身支持 reload 方法）

## 阶段 0：SOP 共创（可选，一次性）

### 场景
用户第一次用 xiaopaw-team、或需要新类型 SOP 时触发。本课 demo 视情况决定是否演示——推荐在开场演示一次，后续项目跑预置 SOP。

### 流程梗概（v1.1 修订：拆成 guide + write 两 skill）

1. 用户飞书："**我们来建一个开发的 SOP 吧**" → Manager kickoff
2. Manager 按 team_protocol 先 read_inbox（无邮件）→ read_project_state（无 project）
3. Manager LLM 看 user_text 含"SOP / 流程 / 标准"，通过 SkillLoader description 匹配到 `sop_cocreate_guide`
4. **第一步（reference）**：`load_skill("sop_cocreate_guide")` → 返回 5 维对话框架 + 结束条件判据给主 Agent 上下文
5. **对话阶段**（主 Agent 用直接 Tool 完成，不再调 skill）：
   - Manager LLM 基于框架，调 `SendToHumanTool(type=info, title="共创 SOP", summary="先问两个基础问题...")` 发出第 1 问
   - 主 Agent 这次 kickoff 结束（ReAct 本轮终止）
   - **用户回复 → FeishuListener → Runner 再次唤醒 Manager → 新一次 kickoff**
   - Manager 本次 kickoff 按 team_protocol：read_inbox（无邮件，因为是飞书常规对话走 `p2p:ou_xxx`）→ 继续上一轮对话（飞书 history 自动注入 backstory）
   - Manager LLM 判断当前进展：如果 5 维未覆盖完 → 继续 `SendToHumanTool` 发下一问；覆盖完 → 进第二步
   - 循环 3-5 轮
6. **第二步（task）**：Manager LLM 判断"5 维覆盖完"，调 `load_skill("sop_write", task_context={sop_name: "feature_dev", 五维摘要: {...}, 阶段清单: [...]})` 
   - Sub-Crew 在沙盒启动，按模板生成完整 SKILL.md，写入 `workspace/manager/skills/sop_{name}/SKILL.md`
   - Sub-Crew 额外调 `workspace/manager/skills/load_skills.yaml` 追加一条 manifest 条目
   - Sub-Crew 返回产出路径给主 Agent
7. **manifest 热重载**：main.py 的 mtime 监听发现 load_skills.yaml 变了 → 触发 Manager 的 SkillLoaderTool `_build_description()` 重跑 → 下次 Manager kickoff 工具 XML 里就能看到新 SOP
8. Manager LLM 调 `SendToHumanTool(type=checkpoint_request, title="新 SOP 草稿 v1", summary=摘要)` 等用户确认
9. 用户 approve → Manager `append_event(action="sop_created", payload={sop_name})` → 结束

### 路由原则（非 hardcoded）
Manager 的 agent.md **没有**"含 SOP 字样就 load sop_cocreate_guide"这种路由表。依赖的是 `sop_cocreate_guide` SKILL 的 `description` 字段里含"建立 SOP 对话框架"这些关键词，SkillLoaderTool 构造的工具 XML 让 LLM 自己做 function calling 匹配。

### 产物
`workspace/manager/skills/sop_feature_dev/SKILL.md`（或 `sop_{name}/`）被固化。manifest 同步更新。后续项目都复用。

## 阶段 1：需求澄清与确认

### 1.1 用户发起

用户："**帮我做个宠物日记小网站，能记录多只宠物，每只宠物有日记时间线。**"

### 1.2 消息路径
- FeishuListener → feishu_bridge.recv_human_response
- 无 pending checkpoint → Runner with routing_key=`p2p:ou_d683...`
- Manager agent_fn（外层 asyncio.Lock 已获取）

### 1.3 Manager kickoff
- 按 team_protocol 元规则：先 read_inbox（无邮件）
- 再 `load_skill("read_project_state")` → 脚本返回"无 active project"
- Manager LLM 看到是"新开发需求"（含"网站/做/小"特征词），通过 SkillLoader description 匹配到 `sop_selector`
- `load_skill("sop_selector")` → 脚本列出所有 kind=sop 的 SKILL（只有一个 sop_feature_dev）→ 匹配命中
- `load_skill("sop_feature_dev")` → 返回主流程 SOP 内容，Manager LLM 进入阶段 1 推进

### 1.4 加载 requirements_guide（reference 型，**v1.1 修订**）

Manager LLM 根据 sop_feature_dev 阶段 1 指令：
1. 先生成 project_id（LLM 决定，如 `proj-pawdiary-{timestamp}`）
2. 调 `CreateProjectTool(project_id)` → mkdir 目录 + 空邮箱 + 空 events.jsonl
3. 调 `AppendEventTool(action="project_created", payload={project_id, sop: sop_feature_dev, user_open_id})`
4. `load_skill("requirements_guide")` ← **reference 型**，返回四维发问框架给主 Agent 上下文

### 1.5 多轮对话阶段（主 Agent 用直接 Tool）

- Manager LLM 基于四维框架，调 `SendToHumanTool(type=info, title="需求澄清", summary="先问两个...")` 发第 1 问
- 本次 kickoff 结束
- **用户回复 → FeishuListener → Runner → Manager 再次 kickoff**
- Manager 按 team_protocol：ReadInbox（无邮件）→ 对话 history 自动注入 backstory（22 课机制）
- Manager LLM 判断当前进展：四维未覆盖完 → 继续 SendToHuman 发下一问；覆盖完 → 进 1.6
- 每轮对话后主 Agent 可选调 `AppendEventTool(action="requirements_q_answered", payload={...})` 记录进展（可选，不做也行）

**关键**：多轮对话期间，主 Agent **不**反复 load requirements_guide（reference skill 返回内容已在上下文里）；主 Agent **不**启动任何 Sub-Crew（没必要）。

### 1.6 对话充分 → 调 requirements_write（task 型，**v1.1 修订**）

Manager LLM 判断"四维都说清楚了"：
- `load_skill("requirements_write", task_context={project_id, 四维摘要: {目标:..., 边界:..., 约束:..., 风险:...}})` 
- Sub-Crew 启动，在沙盒内按模板生成 `requirements.md`，写到 `/workspace/shared/projects/{pid}/needs/requirements.md`
- Sub-Crew 返回路径给主 Agent

主 Agent 收到 Sub-Crew 返回后：
- 调 `AppendEventTool(action="requirements_drafted", artifact="needs/requirements.md")`
- 进 1.7 checkpoint

### 1.7 Checkpoint
Manager LLM 调 `send_to_human(type="checkpoint_request", title="需求文档 v1", summary=摘要)`
- 内部：append_event(action="checkpoint_requested", payload={ck_id, stage: "requirements"})
- 飞书卡片推送

### 1.8 用户确认
用户："确认"
- feishu_bridge.recv_human_response → 找到最近 pending checkpoint → 写 manager.json: `checkpoint_response(ck_id, decision=approved)`
- SendMailTool at cron → Manager 1s 后被唤醒
- Manager kickoff → read_inbox → 看到 approved → append_event(action="checkpoint_approved") → 进阶段 2

## 阶段 2：产品设计（含可选团队评审）

### 2.1 Manager 发 PM 任务
- Manager LLM 按 sop_feature_dev 阶段 2 推进
- `send_mail(to="pm", type_="task_assign", project_id=pid, subject="产品设计", content="读 needs/requirements.md, 按 product_design SKILL 写 design/product_spec.md。自查通过后发 task_done")`
- 内部：at cron 唤醒 PM
- `append_event(action="assigned", to="pm", stage="product_design")`

### 2.2 PM 醒来
- PM agent_fn（routing_key=team:pm, wake_reason=new_mail）
- 按 team_protocol：read_inbox → 看到 task_assign
- PM 的 agent.md 里有"我的决策偏好"元规则，没有"收到 task_assign 去哪个 skill"的路由表
- LLM 读 task_assign 的 content，看到"写产品设计" → SkillLoader 匹配 `product_design`
- 可选：`load_skill("search_memory")` 查历史同类（CRUD / 时间线）的产品经验
- `load_skill("product_design")` task 型，Sub-Crew 启动

### 2.3 product_design SKILL
- 模板字段：概述 / 用户场景 / 功能清单 / 数据模型 / 接口契约 / 交互流程 / 验收标准 / 约束 / 风险
- 每节必填（没内容写"无"）
- 验收标准必须可机械检查（用"应返回 200 / 应匹配正则 / 字段 != null"等关键词，禁用"良好 / 流畅"）
- 自查 checklist 在 SKILL 里，每项附验证命令（如"grep -c '^##' design/product_spec.md ≥ 9"）—— **自查不允许"主观打钩"**，必须机械检查
- 自查通过 → write_shared → send_mail(task_done)

### 2.4 Manager 验收 + 决定是否插评审
- Manager 被唤醒，read_project_state → 看到 stage=product_design assignee=pm
- read_inbox → task_done
- read_shared design/product_spec.md 做基础抽检（节数、关键字段存在）
- **决定是否插评审**：读 team_standards.yaml 的 review_insert_product_design 阈值，比对当前 design 的实体数/接口数。本 demo 3 实体 7 接口 → 刚好触发插评审
- append_event(action="decided_insert_review", reviewers=["rd", "qa"])

### 2.5 并发发 RD/QA 评审
- `send_mail(to="rd", type_="review_request", subject="评审产品文档", content="从 RD 视角评审 design/product_spec.md。按 review_product_design SKILL。输出 reviews/product_design/rd_review.md")`
- 同理 QA
- append_event(action="review_requested", reviewers=["rd","qa"], stage="product_design")

### 2.6 RD 评审（示例）
- RD 醒来 → read_inbox → 看 review_request
- load_skill("review_product_design")（reference）
- 读 `design/product_spec.md`，按 checklist：可实现性 / 性能风险 / 数据一致性 / 接口完整性
- 结论 = pass + 2 个开放技术决策项
- write_shared `reviews/product_design/rd_review.md`（权限：reviews 目录允许写 `{reviewer}_*.md`，这里是 `rd_review.md` 符合）
- `send_mail(to="manager", type_="review_done", content={conclusion: "pass", path: "reviews/product_design/rd_review.md"})`
- `mark_done` 标记自己邮箱里的 review_request 为 done

### 2.7 QA 评审
并行，类似。假设 QA 发现"日期字段时区未规定"→ 结论 = need_changes。

### 2.8 Manager 汇总评审
- Manager 分别被两条 review_done 唤醒（Runner 串行处理）
- 首封 review_done（RD pass）：Manager LLM 按 sop_feature_dev "评审子流程"处理
  - read_project_state 看 pending_reviewers；当前 state 通过"events.jsonl 统计已到 review_done 数"推断
  - 还有 QA 未回 → append_event(action="review_received", reviewer="rd", result="pass") → 继续等
- 次封 review_done（QA need_changes）：
  - 两方都收到 → 评估：QA need_changes → 发回 PM 修
  - `send_mail(to="pm", type_="task_assign", subject="产品设计修订 (第 2 轮)", content="QA 评审发现 2 个问题：...。按 review 修订")`
  - append_event(action="revision_requested", round=2)

### 2.9 PM 修订 + 再评审循环
- 按 team_protocol 约束：subject 带"(第 N 轮)"
- 循环上限 = `team_standards.review_max_rounds`（默认 3）
- 超过 → Manager 发 error_alert 给用户

### 2.10 都 pass 后 → 产品文档 checkpoint
- `send_to_human(type="checkpoint_request", title="产品设计 v2", summary=..., artifacts=["design/product_spec.md", "reviews/product_design/rd_review.md", "reviews/product_design/qa_review.md"])`
- 用户 approve → 进阶段 3

### 6.5 ApplyProposalTool 机械应用（Tool 非 skill，宿主机执行）

**关键修正（v1.0 final）**：apply_proposal 在 v1.0 初稿里同时是 skill 又宣称在宿主机执行 —— 两者互斥。v1.0 final 统一为 **Manager 专属 Python Tool**（见 §11.2），不走 SkillLoader、不进沙盒。

用户逐条 approve 后，Manager LLM 调 `ApplyProposalTool(proposal_id)`，Tool 在主进程内执行（宿主机权限）：
- 读 target file（如 `workspace/pm/skills/product_design/SKILL.md`）当前 SHA256，比对 `checksum_before`
- 不匹配 → 返回 `{errcode: 2, errmsg: "stale, please regenerate proposal"}`
- 匹配 → 机械替换 before → after（正则或精确字符串匹配）
- 自动递增 frontmatter `version`（解析 YAML → v1.0.0 → v1.1.0 → 写回）
- `subprocess.run(["git", "-C", workspace_root, "add", target_path])`
- `subprocess.run(["git", "-C", workspace_root, "commit", "-m", f"[proposal-{proposal_id}] {root_cause}: {title}"])`
- 一条提案一次 commit，便于精准 `git revert`

**workspace/ 必须是独立 git 仓库**（见 §23.3 初始化步骤）。

## 附录 E · 非目标与已知局限

**非目标**：
- 不做 SOP 共创演示外的 SOP 新建（一次演示即可）
- 不做多用户/多团队并发
- 不做档 3 soul/code 级自动 apply
- 不做生产部署 / CI-CD / 前端构建工具链
- 不做多沙盒生产方案（README 说明升级路径）
- 不做 reset_stale（单进程下非必要）
- 不做第二次跑相同任务对比进化效果（git diff 作为进化证据）

**已知局限**：
- 项目并发限 1（同时只有一个 active project，多项目需要扩展 events.jsonl 的项目选择逻辑）
- 飞书 checkpoint 无自动超时（用户长时间不回 → 项目卡住）
- heartbeat 30s 是兜底，正常路径 §10.1 已保证飞书入站与邮件都有 at cron 即时唤醒
- 自我进化的提案质量受 L2/L3 日志信息丰富度影响，首次项目数据少时提案可能偏空
- **asyncio.Lock 释放瞬间的 pgvector 异步索引窗口**：上一轮 Manager 写入 pgvector（`async_index_turn` 是 `asyncio.create_task` 异步触发，不阻塞 kickoff 返回）到下一轮查询有 < 3s 窗口。对本课演示不致命（跨项目检索主要发生在复盘阶段，用户 approve 之间必有等待）
- **events.jsonl 单进程写入**：v1.0 用 filelock 保护单进程内并发，未上 optimistic version。未来若多实例部署需加版本校验（工具层扩展）
- **崩溃半写自愈**：§9.3.2 的自愈检查只能处理 mailbox vs events 不一致；如果 events.jsonl 自身写入过程崩溃（如在写到一半进程死亡），JSON Lines 格式天然容错（最后一行可能部分写入），read_project_state 脚本需跳过非法行并告警
- **review_base 共性未抽取**：PM/RD/QA 的 review_* skill 各自 checklist 有约 40%-60% 共性（读 artifact / 写 review.md / send_mail 回执），v1.0 final 暂保留重复，等实际落地后如维护成本确实高再抽到 `shared/review_base.md`

# XiaoPaw-Team 详细设计文档

> **项目**：xiaopaw-team（模块四第 29 课实战项目）
> **定位**：把 25-28 课的所有机制装配成一个可跑通的数字员工团队，端到端完成一次真实的 toC 项目并自我进化一次
> **技术基座**：`code/xiaopaw-with-memory/`（第 22 课）—— 最大程度复用，尽量不造新轮子
> **版本**：v0.2（根据 v0.1 评审意见重写）
> **最后更新**：2026-04-20

## v0.2 变更说明（相对 v0.1）

v0.1 的几个关键误解已修正：

| # | v0.1 设计 | v0.2 修正 |
|---|---|---|
| 1 | 有"静态 Workflow vs 动态 LLM"分层，Python 编排器（Watchdog / 多 Crew / SOP YAML 解析） | **取消所有 Python 编排**。流程调度完全由 Manager LLM + SOP Skill 驱动，代码只负责：启动 4 个 Agent、路由消息、工具调用、沙盒执行、事件通知 |
| 2 | 每个角色有 N 个 Crew（dialog / assign / review / retro） | **每个角色只有 1 个 Crew**（= 22 课 MainCrew 模式），收到消息后 LLM 靠 SkillLoaderTool 按需加载不同 Skill |
| 3 | 新写 `team_worker.py` / `manager_watchdog.py` 做后台轮询 | **复用 xiaopaw CronService**。每角色注册 1 个 `every 30s` 兜底 cron，发邮件时额外注册 1 个 `at=now+1s` one-shot cron 即时唤醒 |
| 4 | SOP 是结构化 YAML，Python 读 `next_role()` 编排 | **SOP 就是一个 reference 类型的 SKILL.md**，描述"如何处理这类项目"，Manager LLM 读完自己推动流程 |
| 5 | 每角色一个 AIO-Sandbox 容器 | **全团队共享一个 AIO-Sandbox**，用 `/workspace/{role}/` 和 `/workspace/shared/` 目录隔离；WriteSharedFileTool 在工具层强制写权限。生产环境可拆多沙盒，本课示意为主 |
| 6 | 角色间没有显式唤醒接口 | **新增 `notify_role` 工具**（写完邮件后触发）+ Runner 路由扩展支持 `team:{role}` routing_key |
| 7 | Tools / Skills 混淆（有些该是 Skill 的写成了 Tool，反之亦然） | **重新划分**：Tools = 底层原子能力（发邮件、读邮箱、写共享文件、沙盒 bash、飞书发送）；Skills = 业务 SOP（需求澄清、产品设计、代码实现、测试、复盘、sop_feature_dev 主流程） |

其他内容（demo 产品、四层框架、三态邮箱、三层日志、三档 HITL、进化闭环）保持 v0.1 不变。

## 1. 项目定位

xiaopaw-team 是模块四的实战收官项目。它**不是**从头造一个新系统，而是一次**工程装配练习**——把 25-28 课已经讲过的机制组装起来，配一个真实的 toC 项目需求，端到端跑一遍，最后完成一轮团队自我进化。

**与 22 课 xiaopaw-with-memory 的关系**：

| 维度 | 22 课 | 29 课 xiaopaw-team |
|------|------|------|
| Agent 数量 | 1 个（Main Agent + SkillLoaderTool + Sub-Crew） | 4 个（Manager / PM / RD / QA，**每个都是一个独立的 Main Agent**） |
| 人机通道 | 飞书 ↔ Main Agent | 飞书 ↔ **只和 Manager 通信** |
| 团队内协作 | 无 | 文件邮箱（26课三态状态机） |
| Agent 触发 | 飞书消息 → Runner | 飞书消息 / **邮件到达 / cron 兜底** 三种来源都进 Runner |
| 业务流程 | 单次对话 | 端到端项目（需求澄清 → 设计 → 开发 → 测试 → 交付 → 复盘 → 进化） |

**课程价值**：不在于新发明，在于验证"25-28 讲的机制同时在线运行，是否真能做完一个项目"。装配过程中需要补的接缝（唤醒、沙盒共享、单一接口、SOP 作为 skill）就是本课增量。

## 2. 核心设计原则

### 2.1 单一接口原则（27 课保留）

**人永远只和 Manager 沟通**，且本课只有"飞书"这一个人机通道。`send_to_human` 工具只被注入到 Manager Agent，PM / RD / QA 的 Crew 构造器不接受 feishu_sender——这是**架构保证**，不是 LLM 自律。

### 2.2 零 Python 编排，LLM 自驱（v0.2 核心变化）

**代码只做 5 件事**：

1. **启动**：main.py 里构造 4 个 MemoryAwareCrew（对应 4 个角色的 workspace）
2. **路由**：Runner 根据 InboundMessage 的 routing_key 前缀决定进哪个 Agent
3. **工具**：Python 函数暴露为 CrewAI Tool，包含发邮件、读邮箱、写共享文件、飞书发送、沙盒脚本调用、日志钩子
4. **沙盒**：调用 AIO-Sandbox 执行 Skill 脚本（write 文件 / bash 跑命令）
5. **唤醒**：CronService 注册定时/一次性任务，把 wake-up 消息构造成 InboundMessage 进 Runner

**Python 代码里没有任何 "if stage == designing then assign to rd"** 类的分支判断。流程推进全部在 Manager LLM 的对话里完成，依靠 `sop_feature_dev` 这个 reference skill 作为 Manager 的"行动指南"，依靠工具调用作为"执行手段"。

### 2.3 Agent = Crew + SkillLoaderTool + 四层框架

每个角色 = **一个 `MemoryAwareCrew @CrewBase`（复用 22 课）+ 一个 `SkillLoaderTool` + 加若干团队协作工具**。Agent 的 backstory 由 `build_bootstrap_prompt(workspace/{role}/)` 从四层文件（soul / agent / memory / user）拼接而成，与 25 课四层框架完全一致。

Agent 收到任一 InboundMessage（无论来自飞书、邮件通知、cron 兜底）都走同一条路径：kickoff → LLM 读消息 → LLM 决定 load 哪个 Skill → Skill 返回/执行 → LLM 下一步决策。

### 2.4 状态共享 vs 任务触发（26 课保留）

- **共享工作区**（文件系统持久化状态）：`/workspace/shared/projects/{pid}/` 存需求、设计、代码、测试报告。所有角色按目录 Owner 读写
- **文件邮箱**（JSON 结构化事件）：`/workspace/shared/projects/{pid}/mailboxes/{role}.json` 传递 task_assign / task_done / checkpoint_request 等事件
- 邮件里只传路径引用，绝不复制文档全文

### 2.5 不自建调度，复用 xiaopaw CronService（v0.2 核心变化）

xiaopaw 的 CronService 已经实现：`at`（一次性）、`every`（固定间隔）、`cron`（表达式）三种调度模式，触发时构造 InboundMessage 进 Runner 管道。本课**不自建 Worker / Watchdog**，统一用 CronService：

- 每个角色启动时注册 1 个 `every 30s + jitter 5s` 的 cron job，payload 让该角色"检查邮箱"
- 发邮件时由 `notify_role` 工具注册 1 个 `at=now+1s` 的 one-shot，把目标角色唤醒
- 两者对目标 Agent 是同一条路径（都是 Runner 收到 team:{role} 消息）

### 2.6 进化闭环（28 课保留）

一次项目跑完就触发一次复盘。提案严格分档 HITL，evidence 强制非空。本课**只演示到"列出提案 + apply 到 workspace 文件"**，不演示二次执行对比。

## 3. Demo 项目

### 3.1 产品：**"小爪子日记"**（PawDiary）

toC 宠物日记单页网站。选它的理由：
- 呼应 xiaopaw 品牌叙事
- CRUD + 列表 + 时间线，功能量够让 RD 必须写分层代码
- 有可测性（每个接口 pytest）、有前端、有存储
- 一周内做得完

### 3.2 功能清单（MVP）

**实体：**
- `Pet`：id, name, species, birthday, avatar_url, created_at
- `Diary`：id, pet_id, date, content, mood(happy/normal/sad), photos[], created_at

**后端 REST API：**
- Pets：`POST/GET /api/pets`、`GET/PUT/DELETE /api/pets/{id}`
- Diaries：`POST/GET /api/pets/{id}/diaries`、`DELETE /api/diaries/{id}`

**前端：**
- `/`：宠物列表卡片 + 添加按钮
- `/pets/{id}`：单只宠物详情 + 日记时间线 + 新建日记

### 3.3 技术栈是 RD 的自主决定（v0.2 修正）

**技术栈不在设计文档里写死**。`rd/skills/code_impl/SKILL.md` 会包含：

- **推荐栈**（Skill 里列出团队偏好）：后端 FastAPI + SQLAlchemy + SQLite / 前端单文件 HTML+JS+CSS / 测试 pytest
- **硬约束**（Skill 里列出红线）：必须带 pytest 且通过、必须分层（routers/models/schemas）、前端单文件无构建链
- **RD 可以偏离推荐**：如果 RD 判断任务更适合其他方案，允许自主选型，只要满足硬约束

这一点对应用户反馈："具体要看 dev 收到任务如何设计"。

### 3.4 范围边界

- 不做用户系统、不做图片上传到对象存储、不做前端构建工具链、不做部署、不做 CI/CD

### 4.1 整体架构图


### 4.2 关键点

**单进程、单沙盒**：
- 4 个 Agent 都跑在一个 Python 进程里（每个角色一个 `MemoryAwareCrew` 实例，Runner 通过 routing_key 路由）
- 共用一个 AIO-Sandbox 容器（RD 和 QA 需要 bash 跑代码，PM 也在沙盒内写 markdown 以便 RD/QA 能读到）
- Manager 不走沙盒（只做对话、决策、调用内存工具）

**三类消息源统一进 Runner**：
| 来源 | routing_key 示例 | 例子 |
|------|------|------|
| 飞书用户 | `p2p:ou_d683002237...` | 用户发"帮我做个小网站" |
| 邮箱事件（notify_role） | `team:pm` | Manager 发 task_assign 给 PM 后 1 秒触发 |
| 定时兜底 | `team:pm`（同上） | 每 30s PM 被唤醒检查邮箱 |

Runner 对这三类消息的路由和串行队列管理完全一样（xiaopaw 原本机制）。

### 4.3 为什么共享单沙盒

用户反馈"每个角色一个沙盒可能一般机器跑不起来"。本课选择：

- **1 个 AIO-Sandbox 容器**，挂载一个大 `/workspace/` 目录
- 角色间隔离靠**目录前缀 + 工具层权限校验**
- 生产场景可拆（每角色独立容器、独立网络命名空间、独立 API key）
- **本课在 README 里说明这个拆分路径**，不在代码里实现

### 4.4 Agent 实例化

main.py 启动时构造 4 个 Agent：

```python
def build_agents(cfg):
    agents = {}
    for role in ["manager", "pm", "rd", "qa"]:
        workspace_dir = cfg.workspace_root / role
        tools = build_role_tools(role, cfg)   # 每角色工具集不同
        agents[role] = build_agent_fn(workspace_dir, tools)
    return agents

# Runner.dispatch 根据 routing_key 前缀选 agent_fn
```

`build_agent_fn` 是 22 课的现成工厂，接受 workspace + 工具列表，返回 agent_fn。没有新的 Agent 类。

## 5. 唤醒机制

这是 v0.2 最重要的新章节。v0.1 留了空，v0.2 给出完整方案。

### 5.1 设计目标

- 发邮件后**即时**唤醒目标角色（不要等 30s 兜底）
- 避免任何新调度器，只用 xiaopaw 现成的 CronService
- LLM 对"被即时唤醒"和"被定时唤醒"无感——两条路径合一

### 5.2 两条唤醒路径

**A. 兜底轮询（every 30s）**

每个角色启动时，main.py 注册：

```python
cron_service.register_job({
    "id": f"wake-{role}",
    "schedule": {"kind": "every", "interval_sec": 30, "jitter_sec": 5},
    "payload": {
        "routing_key": f"team:{role}",
        "text": "[wake] 定时检查邮箱。请使用 read_inbox 工具查看未处理消息。",
    },
})
```

**B. 即时唤醒（at=now+1s one-shot）**

`notify_role(role)` 工具内部：

```python
def notify_role(role: str):
    cron_service.register_job({
        "id": f"wake-once-{uuid4()}",
        "schedule": {"kind": "at", "timestamp_ms": now_ms() + 1000},
        "payload": {
            "routing_key": f"team:{role}",
            "text": "[wake] 新邮件到达，请立即 read_inbox 处理。",
        },
    })
```

one-shot cron 触发一次后 CronService 自动清理（xiaopaw 原有逻辑）。

### 5.3 两条路径合一

无论哪种触发，CronService 都走同一条路径：
```
CronService._fire → dispatch_fn(InboundMessage) → Runner.dispatch
     → per-routing_key 队列 → agent_fn(text, history, ...)
     → Agent LLM kickoff
```

对 Agent LLM 来说：收到一条系统消息 "[wake]..."，它下一步会调 `read_inbox` 工具拿 unread 列表，再决定做什么。

### 5.4 send_mail 工具联动

```python
def send_mail(to, type_, subject, content, project_id, from_):
    # 1. 写邮箱（26 课 filelock + 三态）
    mailbox_ops.send_mail(mailbox_dir, to, from_, type_, subject, content, project_id)
    # 2. 触发即时唤醒
    notify_role(to)
```

调用方（Manager LLM）不感知唤醒细节，对它就是"一次 send_mail 调用"。

### 5.5 Runner 扩展

xiaopaw 原 Runner 只处理 `p2p:*` / `group:*` / `thread:*`。本课新增 `team:*`：

```python
ROLE_ROUTING_PREFIX = "team:"

def pick_agent_fn(self, routing_key: str):
    if routing_key.startswith(ROLE_ROUTING_PREFIX):
        role = routing_key[len(ROLE_ROUTING_PREFIX):]
        return self._agents[role]
    else:
        return self._agents["manager"]   # 飞书消息默认进 Manager
```

Runner 的 per-routing_key 串行队列天然保证同一角色不会并发 kickoff——这个性质我们直接利用，不额外加锁。

### 5.6 为什么不用 in-process asyncio.Event

更轻量的方案是进程内事件总线。**没选它的原因**：
- 绑定单进程，未来拆多机要重写
- CronService 已经存在且稳定，额外加事件总线增加状态复杂度
- `at cron` 已经就是 "事件 + 延迟"的语义，无需另造概念

### 5.7 未来扩展到多机

生产场景把团队拆到不同机器时：
- `notify_role` 换成 HTTP POST 到目标角色的 URL
- 目标机器上的 Runner 收到同样的 InboundMessage
- 其他所有代码和 Skill 不变

本课不实现多机，但接口设计保留这条路径。

### 8.3 主流程 SOP 作为 Skill 的长啥样

`manager/skills/sop_feature_dev/SKILL.md` 节选：

```markdown
---
name: sop_feature_dev
description: 功能开发项目的标准作业流程。Manager 接到新开发任务时加载此 Skill。
type: reference
version: 1.0.0
---

# SOP: 功能开发流程

你是 Manager。收到用户的新开发需求后，请按以下步骤推动项目。你手上有邮箱工具、
飞书工具、sandbox workspace 工具和 skill 加载工具。

## 阶段 1 · 需求澄清

1. 调 load_skill("requirements_discovery")，按四维框架发问
2. 和用户多轮对话直到需求清晰
3. 调 write_shared 写入 `needs/requirements.md`
4. 调 send_to_human(type=checkpoint, summary="需求 v1 已起草") 等用户确认
5. 用户 approve 前不得进入阶段 2

## 阶段 2 · 产品设计

1. 调 create_project（如果本项目还没初始化）
2. 调 send_mail(to=pm, type=task_assign, subject="产品设计", content="读 needs/..., 
   写 design/product_spec.md")
3. 继续处理其他事务；当收到 PM 的 task_done 时（可能来自 wake_up 或 read_inbox）：
   a. 调 read_shared 读 `design/product_spec.md` 做验收
   b. 若不满足验收标准：发回 PM 并说明问题
   c. 若满足：调 send_to_human(type=checkpoint, summary=设计摘要) 等用户确认
4. 用户 approve 前不进入阶段 3

## 阶段 3 · 代码实现

1. 调 send_mail(to=rd, type=task_assign, subject="代码实现", 
   content="读 design/..., 写 code/ 目录, pytest 必须全 PASS")
2. 收到 RD 的 task_done 后，read_shared 看 code/ 产物
3. 简单抽检（文件是否完整），不做深度评审（交给 QA）
4. 阶段 3 无 checkpoint，直接进阶段 4

## 阶段 4 · 质量验证

1. 调 send_mail(to=qa, type=task_assign, ...)
2. 收到 QA 的 task_done：
   a. 读 test_report.md 看是否有 defects
   b. 无 defects → send_to_human(type=checkpoint, "交付前确认") 等 approve
   c. 有 defects → 判断严重性，必要时发回 RD

## 阶段 5 · 交付

1. 用户 approve 后，send_to_human(type=delivery, 附产物路径 + 启动命令)
2. 进入复盘阶段

## 阶段 6 · 复盘与进化

1. 调 load_skill("team_retrospective")
2. 团队复盘识别瓶颈 → send_mail(to=瓶颈角色, type=retro_trigger)
3. 收到各角色的 retro_proposal 后：
   a. 调 load_skill("review_proposal") 预审
   b. send_to_human(type=proposals_review) 请用户逐条 approve
   c. 用户 approve → load_skill("apply_proposal") 机械应用
4. 应用完成 → send_to_human(type=evolution_done) 总结本次进化

## 重要约束
- 你从不写代码、设计文档、测试用例
- 你从不让 PM/RD/QA 直接联系用户
- 你从不跳过 checkpoint
- 你从不接受无验收标准的需求
```

**这是 v0.2 最关键的设计**：SOP 的流程逻辑全部以自然语言形式写在 SKILL.md 里，Manager LLM 读完就知道整个项目该怎么走。**Python 代码里没有一行 `if stage == X` 的分支**。

### 8.5 Skill 和 Tool 的分工边界

| 判断问题 | 答案 |
|---------|------|
| "发邮件" 是 Skill 还是 Tool？ | **Tool**（单一原子动作） |
| "需求澄清对话"是 Skill 还是 Tool？ | **Skill**（业务流程，涉及多轮决策） |

### 9.2 权限边界（工具层强制）

`WriteSharedTool` 接受 `(project_id, relative_path, content)`，内部：

```python
OWNER_BY_PREFIX = {
    "needs/":  ["manager"],
    "design/": ["pm"],
    "code/":   ["rd"],
    "qa/":     ["qa"],
}

def check_write_permission(role, rel_path):
    for prefix, owners in OWNER_BY_PREFIX.items():
        if rel_path.startswith(prefix):
            if role not in owners:
                raise PermissionError(f"{role} cannot write {rel_path}")
            return
    raise PermissionError(f"unknown path: {rel_path}")
```

这是**架构保证**（Python 层面拒绝），不是 LLM 自律。

## 10. 飞书桥

### 10.1 入（人 → Manager）

FeishuListener 收到用户消息后，进 Runner 之前先过 `feishu_bridge.recv_human_response`：

```python
def recv_human_response(feishu_msg, routing_key):
    # 1. 读 checkpoints.jsonl 找当前 pending 的 checkpoint
    pending = find_pending_checkpoint(routing_key)
    if pending:
        # 2. 写 manager.json：checkpoint_response
        send_mail(to="manager", from_="human", type_="checkpoint_response",
                  project_id=pending["project_id"],
                  content={"checkpoint_id": pending["id"], 
                           "text": feishu_msg.text, 
                           "decision": classify(feishu_msg.text)})
        # 3. 触发 notify_role("manager") 让 Manager 立即处理
        notify_role("manager")
        return "consumed"   # 不再走常规对话
    return None   # 无 pending → 常规对话（新任务 / 澄清回复 / 自由提问）
```

常规对话仍然走 22 课原路径（FeishuListener → Runner → Manager Agent）。

### 10.2 出（Manager → 人）

`SendToHumanTool.call(type, content, artifacts=None)`：

```python
def send_to_human(type_, content, artifacts=None):
    # 1. 写 checkpoints.jsonl 审计
    ck_id = append_checkpoint(project_id, type_, content, artifacts)
    # 2. 构造飞书卡片
    card = build_card(type_, content, artifacts)
    # 3. 发飞书
    FeishuSender.send_card(user_open_id, card)
    # 4. L1 日志
    write_l1_log("manager_to_human", type_, content)
    return ck_id
```

卡片类型：`info` / `checkpoint_request` / `error_alert` / `delivery` / `evolution_report`

### 10.3 人类邮箱在本课变成 checkpoints.jsonl

v0.1 里有 `human.json` 邮箱。v0.2 简化：

- **出向**（Manager→人）：写 `checkpoints.jsonl`（纯审计）+ 发飞书
- **入向**（人→Manager）：飞书消息过桥后直接写 `manager.json`（统一用团队内同一套邮箱）
- `checkpoints.jsonl` 只用于：找 pending checkpoint（按 id 匹配用户回复）+ 审计

好处：邮箱类型少一个，少一套幂等逻辑。

### 10.4 单一接口的架构保证

- `SendToHumanTool` 构造时需要 `feishu_sender` 参数
- `build_role_tools("manager", cfg)` 会注入 `feishu_sender`
- `build_role_tools("pm"/"rd"/"qa", cfg)` **不**传 `feishu_sender`
- PM/RD/QA 的 SkillLoaderTool 里 `send_to_human` 工具根本没注册——LLM 试图调用会报"Tool not found"

## 17. 风险与待确认

### 17.1 已识别工程风险

| # | 风险 | 缓解 |
|---|---|---|
| R1 | Manager 同时被 feishu 消息和 wake 触发 → 可能并发 kickoff | Runner 的 per-routing_key 串行队列天然串行化同一 `p2p:ou_xxx` 或 `team:manager`；但两者 routing_key 不同不会串行 → 用**同 routing_key 策略**：所有 Manager 触发统一走 `team:manager`（飞书消息转发进去）|
| R2 | RD 在沙盒 `pip install` 慢 | 沙盒镜像预装 FastAPI + pytest + httpx + sqlalchemy |
| R3 | RD 自修失败死循环 | `code_impl` SKILL 硬约束 3 次，超出发 error_alert |
| R4 | 用户长时间不回 checkpoint | 本课不做自动超时，在 README 说明 |
| R5 | 多项目并发 | 本课**只支持单项目活跃**，同时只有一个 project_id 在跑 |
| R6 | `team:manager` 和 `p2p:ou_xxx` 两条路径分散了 Manager 状态 | R1 的统一策略顺便解决 |
| R7 | SOP 作为 reference skill，Manager LLM 可能跳步 | `sop_feature_dev` SKILL 内写"每阶段必经 checkpoint"硬约束；用 NEVER 清单强化；失败就 checkpoint 驳回 |

### 17.2 R1 的解决方案详细说明

v0.2 选择把飞书入站消息也路由到 `team:manager`：

```python
# FeishuListener → feishu_bridge.recv_human_response
if not consumed_as_checkpoint:
    # 转化为 team:manager 消息
    inbound = InboundMessage(
        routing_key="team:manager",
        text=feishu_msg.text,
        meta={"source": "feishu", "original_open_id": user_open_id},
    )
    runner.dispatch(inbound)
```

这样 Manager 的所有触发（飞书新消息 / wake / checkpoint_response）都走同一 routing_key，Runner 自动串行化。
